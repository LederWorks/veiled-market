## 1. Decision Summary

The synthesis retains Daedalus (forge), Socrates (probe), and Aristotle (steward) as three
separate plugins — each owns its lifecycle domain — and fills every tooling gap by porting
the copilot monolith's eval loop, JSON schemas, Jinja2 templates, and deterministic validator
into the shared `veiled-market/scripts/` and `veiled-market/templates/` directories, making
those capabilities platform-neutral and available to all current and future trio plugins.
The copilot monolith's `plugin-dev` bash validators are adapted and consolidated into
`veiled-market/scripts/validators/` with Claude-Code-compatible event naming, eliminating
the duplication between platforms. Plugin-dev's agent-creator concept is added to Daedalus
as a new `agent-creator.md` Workflow Agent. Three new Knowledge Skills
(`hook-authoring`, `mcp-authoring`, `settings-authoring`) are appended to Daedalus to close
the domain knowledge gaps. The scaffolder agent is migrated from inline stubs to
`render_template.py` calls so that every future resource is generated from a versioned
Jinja2 template rather than handcrafted per-invocation.

The trio was chosen over a monolith because each plugin's description stays narrowly scoped
(triggering only on its domain's natural-language queries), which minimises Level-1 context
cost — a merged monolith would load all lifecycle verbs into every session unconditionally.

### Lifecycle Chain

```
/forge create → commons/ scaffolded via Jinja2 templates
      ↓
/forge build  → claude/ + copilot/ generated by build-platforms.py
      ↓
/probe        → Socrates scans, writes .todos/socrates/<plugin>-YYYYMMDD.md
      ↓
/steward triage → Aristotle splits auto/manual, suggests /forge remediate
      ↓
/forge remediate → Daedalus applies auto-fixable findings, rebuilds
      ↓
/probe (re-validate) → zero violations? proceed
      ↓
/steward review → Aristotle runs run_loop.py to grade skill descriptions
      ↓
/steward bump   → sync_versions.py propagates semver, finalize.py regen
      ↓
/steward audit  → health-checker fleet sweep, cross-plugin violation check
```

### Fix-Type Routing

```
Socrates TODO finding
   fix-type: edit-content | edit-structure | add-file | delete-file | config-change
        ↓
   Aristotle triage → auto-fixable bucket
        ↓
   /forge remediate → Daedalus applies in priority order (🔴 → 🟡 → 🔵)
        ↓
   rebuild → /probe confirms clean

   fix-type: manual
        ↓
   Aristotle triage → manual bucket → escalate to human
```

### Eval Loop in the Maintenance Cycle

```
/steward review <plugin> <skill>
     ↓
  run_loop.py --skill commons/skills/<s>/SKILL.md --eval-set .evals/<s>.json
     ↓ (per iteration)
  run_eval.py → spawn claude -p per query, detect trigger via stream JSON
     ↓
  improve_description.py → call claude -p with failed queries + history
     ↓ (loop until pass rate ≥ threshold OR max_iterations)
  write best description back to commons/skills/<s>/SKILL.md
     ↓
  /forge build <plugin>
```

#!/usr/bin/env bash
# guard-build-dirs.sh — Block direct writes to generated platform dirs.
# Plugin: daedalus | Event: preToolUse
set -euo pipefail

EVENT=$(cat)
TOOL=$(echo "$EVENT" | jq -r '.tool_name // empty' 2>/dev/null || true)

  *) exit 0 ;;

if echo "${FILE:-}" | grep -qE '/(claude|copilot)/'; then
  echo '{"decision":"block","reason":"Direct writes to claude/ or copilot/ are forbidden. Use /forge build to regenerate platform outputs."}' 
  exit 0
fi
exit 0

## 10. Future-Proofing Decisions

### Decision 1: Shared `veiled-market/scripts/` for eval tooling (not per-plugin)

**Chosen:** `veiled-market/scripts/run_eval.py`, `run_loop.py`, `improve_description.py` —
shared infrastructure invoked by any plugin's Review skill.

**Rejected:** Copying eval scripts into each plugin's `commons/scripts/` (as the copilot
monolith does internally).

**Rationale:** Rule 10a mandates scripts used by 2+ plugins live in shared scripts/. Three
plugins already share `build-platforms.py`. The eval scripts will be used by Aristotle's
review skill across any plugin in the fleet.

**Extension point:** When plugin count reaches 10+, add a `--all` flag to `run_loop.py`
to batch-evaluate all skills in the registry; the shared location makes this trivially
addable without touching any plugin.

---

### Decision 2: Jinja2 templates in `veiled-market/templates/` with `commons/` scope

**Chosen:** All templates produce `commons/` artifacts (platform-neutral). The build system
then generates platform outputs from `commons/`.

**Rejected:** Templates that emit platform-specific (`claude/` or `copilot/`) files directly
(as in the copilot monolith's `scope: "copilot"` templates).

**Rationale:** The `commons/` → `claude/` + `copilot/` build system is the single transform
point. Bypassing it with platform-specific templates would create drift between platforms
and require templates to be updated per platform when a new platform (Codex CLI, Gemini CLI)
is added.

**Extension point:** Adding a third platform (e.g., Gemini CLI) requires only a new
`_build_gemini()` function in `build-platforms.py`. All 38 template-generated plugins remain
untouched.

---

### Decision 3: `preToolUse` hook on Daedalus (not `postToolUse`)

**Chosen:** `preToolUse` on Daedalus that blocks writes to `claude/` or `copilot/`.

**Rejected:** A `postToolUse` hook that detects and reverts writes after they happen.

**Rationale:** Rule 14 is explicit: `preToolUse` MUST be used for guards. `postToolUse` as a
guard is a documented defect. Pre-write interception also avoids partial writes that leave
generated files in an inconsistent state.

**Extension point:** Future lifecycle stages (e.g., a `deploy` step that writes to a registry)
can add additional `preToolUse` guards by appending to the `hooks.json` array without changing
existing hook scripts.

---

### Decision 4: Three Knowledge Skills for hook/mcp/settings (not one combined skill)

**Chosen:** Three separate skills: `hook-authoring`, `mcp-authoring`, `settings-authoring`.

**Rejected:** A single `authoring-guide` skill covering all three domains.

**Rationale:** Rule 3 — every skill addresses exactly one concern. A combined skill would
require the description to cover three distinct trigger scenarios, inflating description word
count and increasing false-positive trigger rate. Each domain has independent natural-language
triggers (`"add a hook"` vs `"configure MCP"` vs `"set model in settings"`).

**Extension point:** A fourth domain (e.g., `lsp-authoring`) is added by creating one new
SKILL.md and one `plugin.base.json` entry — no existing skills are modified.

---

### Decision 5: Agent-creator as a Workflow Agent in Daedalus (not a new plugin)

**Chosen:** `agent-creator.md` as a Workflow Agent in Daedalus, invoked by the `create` skill
when `$RESOURCE_TYPE == "agent"`.

**Rejected:** A standalone `agent-creator` plugin with its own command.

**Rationale:** The agent-creator is a specialist invoked within the `create` workflow, not a
user-facing lifecycle stage. Adding it as a separate plugin would create an 8th entry in the
command namespace for something that fires once per creation session. Workflow agents are
Level-3 resources (zero passive cost) — the right home for specialist sub-tasks.

**Extension point:** Future specialist creators (e.g., `mcp-server-creator`) follow the same
pattern: Workflow Agent in Daedalus, invoked by the `create` skill's dispatch table based on
`$RESOURCE_TYPE`.

---

### Decision 6: Eval set files at `veiled-market/.evals/` (not inside plugins)

**Chosen:** `.evals/<plugin>-<skill>.json` at the marketplace root.

**Rejected:** Eval sets inside each plugin's `commons/` directory.

**Rationale:** Eval sets are marketplace-level test data, not plugin content. They are
maintained by Aristotle's review skill and shared across eval runs. Placing them in `commons/`
would pollute the plugin content tree (which the build system copies verbatim to `claude/`
and `copilot/`), leaking test data into platform outputs.

**Extension point:** A CI pipeline can run `python3 scripts/run_eval.py` for all `.evals/*.json`
files as a regression gate, independent of any single plugin's release cycle.

## 11. Distribution Model: Two-Tier Script Architecture

### The Problem

When a user installs a plugin, only the `copilot/` or `claude/` subdirectory is cached
locally (see MARKETPLACE.md §6.1, §6.3). The marketplace registry confirms this:

```
socrates: source=./plugins/socrates/copilot
daedalus: source=./plugins/daedalus/copilot
aristotle: source=./plugins/aristotle/copilot
```

Scripts at `veiled-market/scripts/` are **not available** after standalone install.
Currently, skills like Daedalus `build` reference `python3 scripts/build-platforms.py` —
this works in-repo but would fail for any user who installs the plugin standalone or
offline.

### The Solution: Two Tiers

**Tier 1 — Fleet Infrastructure (repo-root only, never distributed)**

These scripts are development-time tools that only make sense when you have the full
`veiled-market/` repository. They manipulate the repo itself and are NOT needed by
plugin end-users.

| Script | Location | Reason it stays at root |
|--------|----------|------------------------|
| `build-platforms.py` | `veiled-market/scripts/` | Generates `claude/`+`copilot/` from `commons/` — the build IS the repo |
| `finalize.py` | `veiled-market/scripts/` | Regenerates marketplace.json — repo-level artifact |
| `validate_commons.py` | `veiled-market/scripts/` | Validates `commons/` structure — only exists in-repo |
| `scaffold_plugin.py` | `veiled-market/scripts/` | Creates new plugin directories — repo-level scaffolding |
| `render_template.py` | `veiled-market/scripts/` | Renders Jinja2 templates into `commons/` — build-time tool |
| `sync_versions.py` | `veiled-market/scripts/` | Propagates versions across files — repo-level coordination |
| JSON schemas | `veiled-market/schemas/` | Used by `validate_commons.py` — build-time validation |
| Jinja2 templates | `veiled-market/templates/` | Used by `render_template.py` — build-time scaffolding |

**Tier 2 — Plugin-Local Runtime Scripts (bundled with each plugin)**

These scripts are referenced by skills at runtime and MUST be available when the plugin
is installed standalone. They go into `commons/scripts/` and get copied into `claude/scripts/`
and `copilot/scripts/` by the build system.

| Script | Plugin | Skill that references it |
|--------|--------|--------------------------|
| `run_eval.py` | aristotle | `review` — trigger quality evaluation |
| `run_loop.py` | aristotle | `review` — eval+improve iteration loop |

## 12. Critical Notes

### Security

- Hook scripts MUST NOT read `$HOME/.claude/` or any user credential store. The
  `guard-build-dirs.sh` hook only reads `$COPILOT_TOOL_NAME` and `$COPILOT_TOOL_INPUT`
  from stdin — no filesystem traversal beyond the `file_path` field.
- `run_eval.py` strips the `CLAUDECODE` environment variable before spawning `claude -p`
  subprocesses to prevent terminal conflict (pattern from copilot monolith). Retain this.
- No secrets in templates — hook templates use `$ENV_VAR` references only, never inline values.

### Performance

- Eval runs spawn up to 10 parallel `claude -p` subprocesses by default. On CI with rate
  limits, set `--workers 3`. The `--timeout 120` default should be increased to 180 on
  slow networks.
- `sessionStart` hook (Aristotle) must complete in < 5 seconds. The `sources/plugins.json`
  read + filesystem check for N plugins is O(N) disk ops. At 50 plugins, still < 2s on SSD.
  Add a 5-second `timeoutSec` to the hooks.json entry.

### State Management

- TODO files in `.todos/socrates/` accumulate over time. Aristotle's `triage` skill always
  selects the most-recent file by YYYYMMDD sort. Old files are retained for history — do not
  clean them up automatically. Add a `.todos/` `.gitkeep` to track the directory.
- Eval log files in `.logs/eval/` grow with each `run_loop.py` invocation. Add `.logs/` to
  `.gitignore`; only `.evals/*.json` (test queries) go into git.
- `run_loop.py` writes the best description back to `commons/skills/<s>/SKILL.md` in place.
  Aristotle's Review skill must always rebuild with `/forge build` after a successful loop.

### Migration Steps

1. Run T01–T05 (structural fixes) first — these are zero-risk one-line changes.
2. Run `python3 scripts/build-platforms.py --all` after T05 to confirm existing plugins still
   build cleanly before any new files are added.
3. T08–T16 (templates) can be implemented in any order within Phase 3 because `templates.json`
   has no circular dependencies. Implement `render_template.py` (T16) last so templates can
   be tested immediately.
4. T27 (daedalus plugin.base.json) depends on T34–T36 skill files existing — do not update
   the manifest until the SKILL.md files are in place, or `validate_commons.py` will fail.
5. T30 supersedes T01 (Daedalus hooks.json). Do not apply T01 as a separate step if
   implementing Phase 6 immediately — apply T30 directly.
